1. ABSTRACT
================================================================================

  S-MAS is a high-fidelity satellite operational digital twin that uses
  Multi-Agent Reinforcement Learning (MARL) to autonomously optimize the
  orbital lifetime of a Low Earth Orbit (LEO) microsatellite. The system
  models the ESA PROBA-1 spacecraft (NORAD 26957) at ~600 km altitude,
  integrating real historical space weather data (2001-2020), radiation
  hazard maps, and a validated atmospheric drag model.

  The platform spans four programming languages across a six-phase pipeline:
  C++ (physics engine), Python (MARL training), C# (operational controller),
  and TypeScript/React (WebGPU 3D dashboard). Three heterogeneous AI agents
  (Navigation, Bus Management, Mission) coordinate under a rule-based FDIR
  safety governor using the MAPPO algorithm with Centralized Training and
  Decentralized Execution (CTDE).

  The physics engine was validated against 30 days of real PROBA-1 TLE data
  using SGP4 propagation, achieving an MSE of 336.5 km^2 after calibration.

4. PHASE 1: C++ PHYSICS ENGINE
================================================================================

  4.1 Orbital Mechanics

    - 4th-order Runge-Kutta (RK4) integrator at dt = 5.0 s
    - Earth gravity: point-mass + J2 oblateness perturbation
    - Aerodynamic drag: F_D = -0.5 * rho * Cd * A * v^2 * v_hat
    - Atmosphere co-rotation correction: v_rel = v - (omega x r)
    - Thrust: continuous 3D vector with throttle [0, 1]

  4.2 NRLMSISE-00 Atmospheric Model (Simplified)

    - Exospheric temperature: T_inf = f(F10.7, F10.7_avg, Ap)
    - Bates-Walker temperature profile: T(h) = T_inf - (T_inf - T_120) * exp(-sigma * dh)
    - Diffusive equilibrium density with diurnal and latitudinal modulation
    - density_multiplier = 0.01 (calibrated for multi-year orbital lifetime)
      Short-term validation (30 days): 0.1 gives MSE=336.5 km^2
      Long-term realism (multi-year): 0.01 for realistic ~20+ yr decay at 600km

  4.3 Subsystems Modeled

    - SatelliteBus: GaAs solar arrays (~90 W), Li-ion battery with Arrhenius
      degradation, eclipse/sunlight power cycling
    - Stochastic layer: sensor noise, actuator delay (1-3 steps), actuator
      error (+/-5%), epistemic Cd drift (random walk)
    - SEU generator: probabilistic single-event upsets during SAA transit
    - Ground station visibility: spherical LoS with 5-degree elevation mask

  4.4 C API (DLL Export)

    Function              | Signature
    ----------------------|---------------------------------------------

    smas_init             | (handle) -> int
    smas_reset            | (handle) -> void

    smas_destroy          | (handle) -> void

    ActionPacket: 19 bytes, packed (version 1)

5. PHASE 2: CORE MARL FRAMEWORK
================================================================================

  5.1 Algorithm: MAPPO (Multi-Agent PPO)

    - Shared policy architecture (all agents share weights)
    - Centralized Training, Decentralized Execution (CTDE)
    - GAE (lambda = 0.95, gamma = 0.99) for advantage estimation
    - Clip ratio: 0.2, entropy coefficient: 0.01

  5.2 Observation Space (29 dimensions)

    Category       | Features                              | Dims
    ---------------|---------------------------------------|-----
    Orbital        | alt, lat, lon, |v|, vx_norm, vy_norm  | 6
    Power          | SoC, capacity_frac, solar_w, draw_w   | 4
    Environment    | rho_log, flux10_log, flux30_log, ecl, saa | 5
    Communication  | gs_visible, time_since_contact_norm   | 2
    FDIR           | one-hot [NOM, DEG, SAFE, REC]         | 4
    Degradation    | panel_eff, cd_norm, cycles_norm       | 3
    SEU            | seu_active                            | 1
    Lag features   | kp_3h, f107_3h, kp_6h, f107_6h       | 4

  5.3 Agent Action Spaces

    Agent       | Action Type | Space
    ------------|-------------|-------------------------------
    Navigation  | Continuous  | thrust_xyz [-1,1]^3 + throttle [0,1]
    Bus Manager | Discrete    | deep_sleep {0, 1}
    Mission     | Discrete    | payload_on {0, 1}

  5.4 Survival Reward

    R_t = 1.0 (alive) - 5.0 * dV_used - 2.0 * DoD - 10.0 * |h - 600| - 1000.0 * P_fatal

6. PHASE 3: MISSION LAYER
================================================================================

  6.1 Mission Agent (CHRIS Instrument Control)

    - Toggles optical payload ON/OFF based on orbital position
    - Power draw penalty when active (accelerates discharge)
    - +50.0 reward for valid target imaging
    - -500.0 penalty for payload ON inside SAA boundaries

  6.2 Meta-Coordination

    - If Bus agent activates deep_sleep, payload is forced OFF
    - FDIR governor overrides all agents during DEGRADED/SAFE modes

  6.3 FDIR State Machine

    State      | SoC Trigger | AI Control
    -----------|-------------|----------------------------------
    NOMINAL    | > 20%       | Full autonomy
    DEGRADED   | < 20%       | Payload OFF, thrust capped at 30%
    SAFE       | < 10%       | All overridden, deep sleep forced
    RECOVERY   | > 15%       | Payload OFF, transitioning back

7. PHASE 4: C# CONTROLLER
================================================================================

  7.1 Architecture

    - .NET 10.0 console application
    - P/Invoke bindings to smas_engine.dll with [StructLayout(Pack=1)]
    - ONNX Runtime inference for 3 agent models (nav, bus, mission)
    - Async WebSocket binary server on ws://localhost:8765

  7.2 Binary Telemetry Protocol

    [Version(1B) | Sequence(4B) | PayloadLen(4B) | Payload(98B) | CRC32(4B)]

    - Circular buffer (32 frames) for backpressure management
    - Stochastic network impairment (delay, jitter, packet drops)
    - CSV logger with offline replay engine

  7.3 Fast-Forward Mode (--skip N)

    Simulates all physics steps at full CPU speed, broadcasts only every
    Nth step to the dashboard. Enables multi-year simulations:

      --skip 100    100x speed (1 year in ~1 minute)
      --skip 1000   1000x speed (10 years in ~5 minutes)

    Step count reference:
      1 day = 17,280 | 1 year = 6,307,200 | 10 years = 63,072,000

  7.4 Observation Normalizer

    - Exact C# port of Python's observation.py
    - 29-dim normalized vector (MinMax, Robust, log10, one-hot)

8. PHASE 5: WEBGPU DASHBOARD
================================================================================

  8.1 Technology: Vite + React 19 + TypeScript + Tailwind CSS v4

  8.2 Rendering Engine

    - Primary: WebGPU with WGSL shaders (instanced rendering)
    - Fallback: WebGL2 with GLSL shaders (automatic detection)
    - Both render wireframe Earth (bright cyan-blue) + FDIR-colored satellite
    - Coordinate transform: Lat/Lon/Alt -> ECEF (X, Y, Z)

  8.3 Telemetry Decoder (useTelemetry hook)

    - Zero-allocation DataView parser
    - Directly decodes binary WebSocket stream to React state
    - No JSON parsing overhead
    - Auto-reconnect WebSocket: retries every 2 seconds if controller
      is not yet running (dashboard can be started before controller)

  8.4 HUD Design

    - Deep-space glassmorphism theme (backdrop blur, dark gradients)
    - Dynamic FDIR state indicators with pulse animations
    - Mission Lifetime counter: displays elapsed time as Xy Xd Xh format
    - SATELLITE DECEASED indicator with cause (reentry/battery/comms)
    - Real-time SoC, altitude, agent actions, environment status
    - FDIR-colored satellite: blue (nominal), green (payload ON),
      amber (degraded), red (safe mode)

9. PHASE 6: VALIDATION WITH REAL TLE DATA
================================================================================

  9.1 Methodology

    - Parsed 43,000+ historical PROBA-1 TLEs from Space-Track
    - Used Python sgp4 library to establish ground-truth altitude curve
    - Ran S-MAS engine (zero thrust) for equivalent 30-day window
    - Compared simulated vs real decay profiles

  9.2 Density Multiplier Calibration

    Multiplier | MSE (km^2) | Use Case
    -----------|------------|-----------------------------
    0.01x      | N/A        | Long-term realism (20+ yr)
    0.10x      | 336.5      | Best 30-day accuracy
    0.25x      | 438.4      | -
    0.50x      | 619.3      | -
    0.75x      | 860.5      | -
    1.00x      | 1145.3     | Raw model (overestimates)

  9.3 Dual Calibration Strategy

    The simplified NRLMSISE-00 model overestimates thermospheric density
    by approximately 10-100x at 600 km. Two calibration points are used:

    - Short-term validation (30 days): multiplier = 0.1
      RMSE ~18.3 km, suitable for training and unit tests.

    - Long-term realism (multi-year): multiplier = 0.01
      Matches real PROBA-1 behavior (24-year operational life at ~600 km).
      Set as the default for operational simulations.

11. KNOWN LIMITATIONS
================================================================================

  1. Atmospheric model is a simplified NRLMSISE-00 approximation, not the
     full empirical model. Uses dual calibration (0.1 for short-term,
     0.01 for long-term realism).
  2. Training convergence requires substantial compute (>1M steps for
     meaningful policy improvement). Current agents exhibit orbit-raising
     behavior that reduces long-term stability.
  3. Single-satellite simulation. Swarm scaling (N>1) requires extending
     the observation space and action coordination protocol.
  4. WebGPU support requires modern browsers (Chrome/Edge 113+).
  5. The C++ engine does not include SGP4 propagation internally; it uses
     Keplerian initialization + numerical RK4 integration.
  6. The modular algorithm design (GUIDE.md) allows swapping MAPPO for
     MADDPG, QMIX, IPPO, HAPPO, or other MARL algorithms by editing
     only 2 files (mappo.py + train.py imports).
